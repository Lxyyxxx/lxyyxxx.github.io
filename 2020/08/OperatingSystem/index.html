# 操作系统引论

名词解释

# 操作系统

配置在计算机硬件上的第一层软件，是对硬件系统的首次扩充，其主要作用是管理好这些设备，提高他们的利用率和系统的吞吐量，并为用户和应用程序提供一个简单的接口，便于用户使用

# 分时系统

在一台主机上连接了多个配有显示器和键盘的终端并由此所组成的系统，该系统允许多个用户同时通过自己的终端，以交互方式使用计算机，共享主机中的资源

# 实时系统

系统能及时响应外部事件的请求，在规定的时间内完成对该事件的处理，并控制所有实时任务协调一致地运行

简答

# 分时系统和实时系统的区别与联系

特征 分时系统 实时系统
多路性 系统按分时原则为多个终端用户服务 系统周期性地对多路现场信息进行采集，以及对多个对象或多个执行机构进行控制
独立性 每个终端用户在与系统交互时，彼此相互独立互不干扰 对信息的采集和对对象的控制彼此互不干扰
及时性 依据人能接收的等待时间确定 以控制对象所要求的截止时间确定
交互性 仅限于访问系统中某些特定的专用服务程序 能向终端用户提供数据处理、资源共享等服务
可靠性 要求系统可靠 要求系统高度可靠

# 操作系统的特性

并发和共享是两个最基本的特征，又是互为存在的条件

# 1. 并发

并行：两个或多个事件在同一时刻发生
并发：两个或多个事件在同一时间间隔发生，在一段时间内，宏观上有多个程序在同时运行，但微观上只能分时交替执行

# 2. 共享

资源服用，系统中的资源可供内存中多个并发执行的进程共同使用

# 3. 虚拟

将一个物理实体变为若干个逻辑上的对应物
通过时分复用、空分复用实现虚拟

# 4. 异步

进程是以人们不可预知的速度向前推进的

# 操作系统的主要功能

# 1. 处理机管理功能

作业、进程

  1. 进程控制：创建和撤销进程
  2. 进程同步：对诸进程的运行进行协调
  3. 进程通信：实现进程之间的信息交换
  4. 调度：按照一定算法把处理机分配给进程，包括作业调度和进程调度两步

# 2. 存储器管理功能

内存

  1. 内存分配和回收：静态、动态分配，作业装入时确定内存空间，动态允许作业在运行过程中继续申请新的附加内存空间
  2. 内存保护：确保每道用户程序仅在自己的内存空间运行，彼此互不干扰；绝不允许用户访问操作系统中的程序和数据，也不允许用户程序转移到****非共享的其他用户程序中去执行
  3. 地址映射：将地址空间中的逻辑地址转为内存空间中的物理地址
  4. 内存扩充：使用虚拟存储技术，从逻辑上扩充，有请求调入和置换功能

# 3. 设备管理功能

I/O

  1. 缓冲管理：缓和CPU和I/O设备速度不匹配的矛盾
  2. 设备分配：根据用户的I/O请求、系统现有资源情况，按照某种设备分配策略，为之分配设备
  3. 设备处理：驱动，实现CPU和设备管理器间的通信

# 4. 文件管理功能

用户文件、系统文件、外存空间

  1. 文件存储空间的管理：分配、回收存储空间，提高外存利用率从而提高文件的存取速度
  2. 目录管理：为每个文件建立一个目录项，实现目录共享、快速检索
  3. 文件的读/写管理和保护：读/写指针；防止未授权用户存取文件、防止冒名顶替存储文件、防止以不正确的方式使用文件

# 5. 提供用户与操作系统的接口

  1. 用户接口：便于用户直接或间接地控制自己的作业，分为联机、脱机、图形
  2. 程序接口：为用户程序在执行中访问系统资源而设置的，是用户程序取得操作系统服务的唯一途径

# 1. 动态性

进程的最基本特征
随创建而产生，由调度而执行，由撤销而消亡

# 2. 并发性

多个进程实体同存于内存中，且能在一段时间内同时运行

# 4. 异步性

进程实体按异步方式运行，即进程按各自独立的、不可预知的速度向前推进

# 进程同步原则

# 1. 空闲让进

当无进程处于临界区时，表明临界资源处于空闲状态，应允许一个请求进入临界区的进程立即进入自己的临界区，以有效地利用临界资源

# 2. 忙则等待

当已有进程进入临界区时，表明临界资源正在被访问，因而其它试图进入临界区的进程必须等待，以保证对临界资源的互斥访问

# 3. 有限等待

对要求访问临界资源的进程，应保证在有限时间内能进入自己的临界区，以免陷入“死等”状态

# 4. 让权等待

当进程不能进入自己的临界区时，应立即释放处理机，以免进程陷入“忙等”状态

# 进程和线程的区别

# 1. 调度的基本单位

在引入线程的操作系统中，则把线程作为调度和分派的基本单位，而进程作为资源拥有的基本单位

# 2. 并发性

在引入线程的操作系统中，不仅进程之间可以并发执行，而且在一个进程中的多个线程之间亦可并发执行

# 3. 拥有资源

一般而言，线程自己不拥有系统资源(也有一点必不可少的资源)，但它可以访问其隶属进程的资源

# 4. 独立性

在同一进程中不同线程之间的独立性要比不同进程中间的独立性低很多
不同进程有独立的存储地址空间和资源，不允许其他进程访问
同一进程的不同线程共享进程的存储地址空间和资源

# 5. 系统开销

进程切换时，涉及到当前进程CPU环境的保存及新被调度运行进程的CPU环境的设置
线程的切换则仅需保存和设置少量寄存器内容，不涉及存储器管理方面的操作
由于一个进程中的多个线程具有相同的地址空间，在同步和通信的实现方面线程也比进程容易
在一些操作系统中，线程的切换、同步和通信都无须操作系统内核的干预

# 6. 支持多处理系统

单线程进程只能在一个处理机上运行
多线程进程可以将不同线程分配到多个处理机上运行

名词解释

# 周转时间

从作业被提交给系统开始，到作业完成为止的这段时间间隔
包括四部分时间：

  1. 作业在外存后备队列上等待(作业)调度的时间
  2. 进程在就绪队列上等待进程调度的时间
  3. 进程在CPU上执行的时间
  4. 进程等待I/O操作完成的时间

# 响应时间

从用户通过键盘提交一个请求开始，直至系统首次产生响应为止的时间，或者说，直到屏幕上显示出结果为止的一段时间间隔

# 死锁

一组进程中的每一个进程都在等待仅由该进程中的其他进程才能引发的事件，那么该组进程是死锁的
多个进程在运行过程中因争夺资源而造成的一种僵局，当进程出于这种僵持状态时，若无外力作用，它们将无法再向前推进

# 安全状态

系统能按某种进程推进顺序(P1,P2,…,Pn)(P_1,P_2,\dots,P_n)(P1​,P2​,…,Pn​)为每个进程PiP_iPi​分配其所需资源，直至满足每个进程对资源的最大需求，使每个进程都可以顺利地完成
安全序列：(P1,P2,…,Pn)(P_1,P_2,\dots,P_n)(P1​,P2​,…,Pn​)

# 基于时间片的轮转调度算法 RR

系统将所有的就绪进程按先来先服务的原则排成一个队列，每次调度时，把CPU分配给队首进程，并令其执行一个时间片。当执行的时间片用完时，由一个计时器发出时钟中断请求，调度程序便据此信号来停止该进程的执行，并将它送往就绪队列的末尾，然后，再把处理机分配给就绪队列中新的队首进程，同时也让它执行一个时间片

# 多级反馈队列调度算法

设置多个就绪队列，并为各个队列赋予不同的优先级。第一个队列的优先级最高，第二个队列次之，其余各队列的优先权逐个降低
该算法赋予各个队列中进程执行时间片的大小也各不相同，在优先权愈高的队列中，为每个进程所规定的执行时间片就愈小

# 最高优先权优先 FPF

把处理机分配给就绪队列中优先权最高的进程

  1. 非抢占式优先权算法：系统一旦把处理机分配给就绪队列中优先权最高的进程后，该进程便一直执行下去，直至完成；因发生某事件使该进程放弃处理机时，系统方可再将处理机重新分配给另一优先权最高的进程
  2. 抢占式优先权调度算法：只要又出现了另一个其优先权更高的进程，进程调度程序就立即停止当前进程的执行，重新将处理机分配给新到的优先权最高的进程

# 产生死锁的必要条件

必须同时具备下面四个必要条件，死锁才会发生

  1. 互斥条件：一段时间内，某资源只能被一个进程占用
  2. 请求和保持条件：进程保持了一个资源，请求新的资源，但该资源被其他资源占用，请求进程被阻塞，但对自已已获得的资源保持不放
  3. 不可抢占条件：进程已获得的资源在未使用完之前不能被抢占，只能在进程使用完时由自己释放
  4. 循环等待条件：在发生死锁时，必然存在一个进程——资源的环形链

# 处理死锁的基本方法

# 1. 预防死锁

破坏四个必要条件中的一个或几个

# 破坏请求和保持条件

  1. 第一种协议：一次性申请所有资源，不会再提出请求资源的要求，破坏了请求条件；分配资源时，如果有一种资源不满足进程的要求，全部资源都不分配，进程在等待时未占用任何资源，破坏了保持条件
  2. 第二种协议：一个进程只获得运行初期所需资源后，便开始运行，运行过程中再逐步释放已分配给自己的且用完的全部资源，再请求新的资源

# 破坏不可抢占条件

提出新的资源请求不能得到满足时，必须释放已保持的所有资源

# 破坏循环等待条件

规定每个进程必须按资源序号递增的顺序请求资源，需要多个同类资源必须一起请求，请求低序号的资源必须释放已保持的高序号的资源

# 2. 避免死锁

在资源的动态分配过程中，用某种方法去防止系统进入不安全状态
银行家算法

/* 数据结构 */
int m;                // 资源数
int n;                // 进程数
int Available[M];     // 可利用资源向量
int Max[N][M];        // 最大需求矩阵
int Allocation[N][M]; // 分配矩阵
int Need[N][M];       // 需求矩阵
// Need + Allocation = Max

/* 银行家算法 */
bool safe();
void print();
void banker(int i) { // 进程i
    int Request[M]; // 进程i的请求向量
    cout << "Request：";
    for (int j = 0; j < m; j++) cin >> Request[j];
    // 对每个资源进行检查
    for(int j=0; j<m; j++) {
        // 1 边界判断
        if(Request[j]>Need[i][j]) {
            printf("ERROR: 所需要的资源数已超过所宣布的最大值\n");
            break;
        }
        // 2 边界判断
        if(Request[j]>Available[j]) {
            printf("ERROR: 尚无足够资源\n");
            break;
        }
        // 3 试探分配
        Available[j]-=Request[j];
        Allocation[i][j]+=Request[j];
        Need[i][j]-=Request[j];
        // 4 安全性算法
        // 不安全，撤销分配
        if(!safe()) {
            Available[j]+=Request[j];
            Allocation[i][j]-=Request[j];
            Need[i][j]+=Request[j];
            printf("ERROR: 系统不安全\n");
            break;
        }
    }
    // 安全，输出安全序列
}

/* 安全性检查 */
int SafeOrder[N]; // 安全序列
bool safe() {
    int cur=0; // 安全序列下标
    // 1 初始化工作向量Work和结束向量Finish
    int Work[M];
    int Finish[N];
    for(int j=0; j<m; j++)
        Work[j]=Available[j];
    for(int i=0; i<n; i++)
        Finish[i]=false;
    while(1) {
        // 2 从进程集合中找到满足下列条件的进程
        // Finish[i]=false
        // Need[i][j]<=Work[j]
        int i;
        for(i=0; i<n; i++) {
            if(Finish[i])
                continue;
            bool f=true;
            for(int j=0; j<m; j++) {
                if(Need[i][j]>Work[j]) {
                    f=false;
                    break;
                }
            }
            // 找到了
            if(f)
                break;
        }
        // 3 找到符合条件的进程了，代表该进程可以顺利完成，释放它的资源，继续循环
        if(i<n) {
            for(int j=0; j<m; j++) {
                Work[j]+=Allocation[i][j];
                Finish[i]=true;
            }
            SafeOrder[cur++]=i; // 存入安全序列
        }
        // 找不到符合条件的进程了
        else
            break;
    }
    // 4 找不到符合条件的进程了，判断是否所有Finish都为true
    for(int i=0; i<n; i++)
        if(!Finish[i])
            return false;
    return true;
}

# 程序的装入方式

将单个目标模块/装入模块装入内存

  1. 绝对装入方式：装入到内存中事先指定的位置，不进行地址的转换，相对地址/逻辑地址和实际内存中的地址完全相同

# 动态分区分配

数据结构：

  1. 空闲分区表：记录每个空闲分区的情况
  2. 空闲分区链：将所有空闲分区链接成一个双向链

算法：
按照分配算法从空闲分区表/链中选择一个分区分配给该作业

分配内存：
从空闲分区表/链中找到所需大小的分区，若多余部分太小，则不再切割，将整个分区分配给请求者；否则，将余下部分留在空闲分区表/链中

回收内存：

  1. 回收区和插入点的前一个空闲分区F1F_1F1​相邻接，将回收区和插入点的前一分区合并，修改前一分区F1F_1F1​的大小
  2. 回收区和插入点的后一个空闲分区F2F_2F2​相邻接，将回收区和插入点的后一分区合并，修改回收区F1F_1F1​的大小，将回收区的首址作为新空闲区的首址
  3. 回收区同时与插入点的前后两个分区邻接，将三个分区合并，修改前一分区F1F_1F1​的大小，取消F2F_2F2​的表项

# 基于顺序搜索的动态分区分配算法

  1. 首次适应FF：每次从链首开始找第一个大小能满足的空闲分区
  2. 循环首次适应NF：每次从下一个空闲分区开始查找第一个大小能满足的空闲分区
  3. 最佳适应BF：总是把能满足要求又是最小的空闲分区分配给作业
  4. 最坏适应WF：总是挑选最大的空闲分区分配给作业

在选完空闲分区后，需要把相应大小的分区分配给作业，剩下的留在空闲分区链中

# 基于索引搜索的动态分区分配算法

  1. 快速适应QF：根据容量分类，对每一类具有相同容量的所有空闲分区单独设一个空闲分区链表
  2. 伙伴系统：每个分区大小为2k,k∈[1,m]2^k,k\in[1,m]2k,k∈[1,m]，要找2i2^i2i的空闲分区，需要对2i+n2^{i+n}2i+n的空闲分区进行nnn次分割得到2i2^i2i的空闲分区
  3. 哈希算法：以空闲分区大小为关键字的哈希表，每个表项为对应空间分区链表的表头指针

# 程序的局部性原理

在一较短时间内，程序的执行仅局限于某个部分，它所访问的存储空间也局限于某个区域

  1. 程序执行时，除了少部分的转移和过程调用指令外，在大多数情况下仍是顺序执行的
  2. 过程调用将会使程序的执行轨迹由一部分区域转至另一部分区域，但经研究看出，过程调用的深度在大多数情况下都不超过5，即程序将会在一段时间内都局限在这些过程的范围内运行
  3. 程序中存在许多循环结构，这些虽然只由少数指令构成，但是它们将多次执行
  4. 程序中还包括许多对数据结构的处理，如对数组进行操作，它们往往都局限于很小的范围内

局限性还表现在下述两个方面：

  1. 时间局限性：如果程序中的某条指令一旦执行，则不久以后该指令可能再次执行；如果某数据被访问过，则不久以后该数据可能再次被访问。产生时间局限性的典型原因是由于在程序中存在着大量的循环操作
  2. 空间局限性：一旦程序访问了某个存储单元，在不久之后，其附近的存储单元也将被访问，即程序在一段时间内所访问的地址，可能集中在一定的范围之内，其典型情况便是程序的顺序执行

# 物理块的分配策略

固定分配：为每个进程分配一组固定数目的物理块，在进程运行期间不再改变
可变分配：为每个进程分配一组固定数目的物理块，在进程运行期间可根据情况做适当的增加或减少
局部置换：如果进程在运行中发生缺页，则只能从分配给该进程的n个页面选出一页换出，然后再调入一页
全局置换：如果进程在运行中发生缺页，则将OS所保留的空闲物理块取出一块分给该进程，或者以所有进程的全部物理块为标的，选择一块换出，然后将所缺之页调入

# 页面置换算法

  1. 最佳置换算法Optimal：被淘汰的页面是以后永不使用的或是最长（未来）时间内不再被访问的页面
  2. 先进先出页面置换算法FIFO：总是淘汰最先进入内存的页面，即选择在内存中驻留时间最久的页面予以淘汰
  3. 最近最久未使用置换算法LRU：选择最近最久未使用的页面予以淘汰
  4. 最少使用置换算法LFU：选择最近时期使用最少的页面作为淘汰页
  5. Clock置换算法/最近未用算法NRU：选择第一个访问位为0（代表未使用过）的页面换出去
  6. 改进型Clock置换算法：考虑页面的使用情况（访问位A）和置换代价（修改位M），优先选择A=0,M=0、A=0,M=1、A=1,M=0、A=1 M=1进行替换

# 设备分配方式

  1. 安全分配方式：每当进程发出I/O请求后，便进入堵塞状态，直到其I/O操作完成时才被唤醒。破坏了请求和保持条件，不会造成死锁
  2. 不安全分配方式：进程在发出I/O请求后仍继续运行，需要时又发出第二个I/O请求、第三个I/O请求等，仅当进程所请求的设备已被另一进程占用时，才进入堵塞状态。可能具备请求和保持条件，造成死锁
